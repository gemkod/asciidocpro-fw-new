= Logique d'héritage des configurations
:toc: left
:toclevels: 3
:icons: font

== Principe général

Le framework applique un système d'héritage à quatre niveaux pour la configuration des attributs. Chaque niveau peut surcharger les valeurs du niveau précédent :

[source]
----
   Système (défaut)
       │
       ▼
   Global (surcharge optionnelle — concerne tous les projets)
       │
       ▼
   Projet (surcharge optionnelle)
       │
       ▼
   Document (surcharge optionnelle)
       │
       ▼
   Resolver (résolution finale)
       │
       ▼
   Alias (exposition utilisateur)
----

L'idée est simple : **le dernier qui parle a raison**. En AsciiDoc, quand un attribut est redéfini, sa nouvelle valeur écrase silencieusement l'ancienne. Le framework exploite ce comportement en chargeant les fichiers de configuration dans un ordre précis.

== Les quatre niveaux de configuration

=== Niveau 1 — Système (valeurs par défaut)

[cols="1,3"]
|===
| Fichier | `system/attributes/attributes_config.adoc`
| Chargé | Toujours, inconditionnellement
| Rôle | Définit **toutes** les valeurs par défaut du framework
|===

Ce fichier contient l'intégralité des attributs configurables avec leurs valeurs par défaut. C'est la référence complète. Exemples :

[source,asciidoc]
----
:__render_show_correction: 1
:__theme_pdf_vendor: asciidocpro
:__theme_pdf_file_name: adp-theme.yml
:__project_theme_enabled: 0
:__document_theme_enabled: 0
:__project_docinfo_enable: 0
:__document_docinfo_enable: 0
----

A ce stade, le framework utilise le thème `asciidocpro` avec le fichier `adp-theme.yml`, et les surcharges global/projet/document sont désactivées.

=== Niveau 2 — Global (surcharge optionnelle)

Le niveau global se situe entre le système et le projet. Il permet de redéfinir des attributs **sans jamais toucher à la configuration système**, et ces redéfinitions s'appliquent alors à **tous les projets**. Chaque projet peut ensuite surcharger à son propre niveau.

Le dossier global se trouve à la racine du framework :

[source]
----
framework/
└── global/
    ├── attributes/
    │   ├── enable_config.adoc
    │   ├── attributes_config.adoc
    │   └── attributes_content.adoc
    ├── docinfo/
    ├── templates/
    └── themes/
----

Le mécanisme de chargement est identique à celui du projet : verrou + chargement conditionnel.

==== Etape 1 : le verrou (`enable_config.adoc`)

[source,asciidoc]
----
// loader.adoc
:__configuration_enabled: 0            <1>
:__attributes_content_enabled: 0
include::{__global_attributes_enable_config_file}[]   <2>
----
<1> Le verrou est **réinitialisé à 0** avant le niveau global
<2> Le fichier `enable_config.adoc` du global est inclus

==== Etape 2 : chargement conditionnel

[source,asciidoc]
----
ifeval::[{__configuration_enabled} == 1]
include::{__global_attributes_config_file}[]
endif::[]
ifeval::[{__attributes_content_enabled} == 1]
include::{__global_attributes_content_file}[]
endif::[]
----

Si le verrou vaut `1`, le fichier `attributes_config.adoc` du global est chargé. Ses attributs **écrasent** ceux du système, et s'appliqueront à tous les projets (sauf si un projet les surcharge à son tour).

==== Exemple concret

Si le fichier `global/attributes/attributes_config.adoc` contient :

[source,asciidoc]
----
:__global_theme_enabled: 1
:__theme_pdf_vendor: mon_organisation
:__theme_pdf_file_name: organisation-theme.yml
----

Tous les projets utiliseront par défaut le thème `mon_organisation/organisation-theme.yml` au lieu du thème système, sans que les fichiers système n'aient été modifiés. Un projet peut ensuite redéfinir son propre thème s'il le souhaite.

=== Niveau 3 — Projet (surcharge optionnelle)

Le chargement du niveau projet se fait en deux temps via un mécanisme de verrou :

==== Etape 1 : le verrou (`enable_config.adoc`)

[source,asciidoc]
----
// loader.adoc, lignes 13-14
:__configuration_enabled: 0            <1>
include::{__project_attributes_enable_config_file}[]   <2>
----
<1> Le verrou est **réinitialisé à 0** avant chaque niveau
<2> Le fichier `enable_config.adoc` du projet est inclus

Le fichier `enable_config.adoc` du projet contient simplement :

[source,asciidoc]
----
:__configuration_enabled: 0    // ou 1 pour activer
----

==== Etape 2 : chargement conditionnel

[source,asciidoc]
----
// loader.adoc, lignes 15-18
ifeval::[{__configuration_enabled} == 1]
include::{__project_attributes_config_file}[]
endif::[]
----

Si le verrou vaut `1`, le fichier `attributes_config.adoc` du projet est chargé. Ses attributs **écrasent** ceux du système.

==== Exemple concret

Le projet `00000000` a `__configuration_enabled: 0` donc sa config n'est pas chargée. Mais s'il avait `1`, son fichier `attributes_config.adoc` :

[source,asciidoc]
----
:__project_theme_enabled: 1
:__theme_pdf_vendor: er
:__theme_pdf_file_name: er-theme.yml
:__project_docinfo_enable: 0
:__project_docinfo_vendor: er
----

...viendrait surcharger les valeurs système ou globales : le thème passerait à `er/er-theme.yml`.

=== Niveau 4 — Document (surcharge optionnelle)

Le mécanisme est identique au niveau projet :

[source,asciidoc]
----
// loader.adoc, lignes 20-25
:__configuration_enabled: 0                              <1>
include::{__document_attributes_enable_config_file}[]    <2>
ifeval::[{__configuration_enabled} == 1]
include::{__document_attributes_config_file}[]           <3>
endif::[]
----
<1> Réinitialisation du verrou (indépendant du niveau projet)
<2> Le fichier `enable_config.adoc` du document active ou non le chargement
<3> Si activé, les attributs du document écrasent tout ce qui précède

==== Exemple concret

Le document `11111111` a `__configuration_enabled: 1`, donc son fichier est chargé :

[source,asciidoc]
----
:__render_question_start: 56
:__document_theme_enabled: 1
:__theme_pdf_vendor: ersn
:__theme_pdf_file_name: ersn-theme.yml
----

Après ce chargement, le thème est maintenant `ersn/ersn-theme.yml` — ni celui du système, ni celui du projet.

== Résumé du mécanisme d'écrasement

Le tableau suivant illustre la cascade pour l'attribut `:__theme_pdf_vendor:` à travers les quatre niveaux :

[cols="1,2,2,2"]
|===
| Etape | Fichier chargé | Valeur de `__theme_pdf_vendor` | Explication

| 1 | `system/.../attributes_config.adoc`
| `asciidocpro`
| Valeur par défaut du système

| 2 | `global/.../attributes_config.adoc`
| `mon_organisation`
| Le global surcharge pour tous les projets

| 3 | `project/.../attributes_config.adoc`
| `er`
| Le projet surcharge avec son propre thème

| 4 | `document/.../attributes_config.adoc`
| `ersn`
| Le document surcharge avec son propre thème

| _Final_ | _(valeur utilisée)_
| `ersn`
| Le dernier niveau activé l'emporte
|===

== Phase de résolution

Après le chargement des quatre niveaux, le fichier `attributes_resolver.adoc` transforme les attributs internes (`__xxx`) en attributs natifs AsciiDoc. Le resolver applique lui aussi une logique de cascade pour certains attributs.

=== Exemple : le thème PDF

[source,asciidoc]
----
// resolver
:pdf-themesdir: {__system_themes_folder}{__theme_pdf_vendor}/     <1>
ifeval::[{__global_theme_enabled} == 1]
:pdf-themesdir: {__global_themes_folder}{__theme_pdf_vendor}/     <2>
endif::[]
ifeval::[{__project_theme_enabled} == 1]
:pdf-themesdir: {__project_themes_folder}{__theme_pdf_vendor}/    <3>
endif::[]
ifeval::[{__document_theme_enabled} == 1]
:pdf-themesdir: {__document_themes_folder}{__theme_pdf_vendor}/   <4>
endif::[]
:pdf-theme: {__theme_pdf_file_name}
----
<1> Par défaut : cherche le thème dans `system/themes/`
<2> Si le global active son thème : cherche dans `global/themes/`
<3> Si le projet active son thème : cherche dans `projects/{id}/themes/`
<4> Si le document active son thème : cherche dans `projects/{id}/documents/{did}/themes/`

Le resolver utilise la **même valeur** de `__theme_pdf_vendor` (déjà surchargée par la cascade) mais change le **dossier de base** selon le niveau activé. Ce double mécanisme permet de :

- Utiliser un thème système existant depuis un dossier global ou projet (changer le dossier, garder le vendor)
- Utiliser un thème complètement différent (changer les deux)

=== Même logique pour le docinfo

[source,asciidoc]
----
// resolver
ifeval::[{__docinfo_enable} == 1]
:docinfodir: {__system_docinfo_folder}{__system_docinfo_vendor}/
ifeval::[{__global_docinfo_enable} == 1]
:docinfodir: {__global_docinfo_folder}{__global_docinfo_vendor}/
endif::[]
ifeval::[{__project_docinfo_enable} == 1]
:docinfodir: {__project_docinfo_folder}{__project_docinfo_vendor}/
endif::[]
ifeval::[{__document_docinfo_enable} == 1]
:docinfodir: {__document_docinfo_folder}{__document_docinfo_vendor}/
endif::[]
endif::[]
----

=== Héritage des templates

Les templates sont des fragments AsciiDoc réutilisables qui définissent la mise en forme de certains éléments récurrents (labels de question, de réponse, de note enseignant, etc.). Leur héritage fonctionne sur le même principe de cascade, mais avec un mécanisme propre.

==== Les 5 templates du framework

Le système fournit 5 templates par défaut dans `system/templates/` :

[cols="2,3"]
|===
| Fichier | Rôle

| `label_question_template.adoc`
| Mise en forme du label de question (ex : *Q 1)*)

| `label_answer_template.adoc`
| Mise en forme du label de correction (ex : _Correction de 1_)

| `label_answer_available_template.adoc`
| Message affiché quand la correction est masquée (ex : _La réponse à la question *1* est disponible dans la correction._)

| `label_teacher_note_template.adoc`
| Mise en forme du label de note enseignant

| `label_todo_template.adoc`
| Mise en forme du label de tâche à faire
|===

==== Mécanisme de cascade

Contrairement au thème et au docinfo qui utilisent un flag unique par niveau (`__global_theme_enabled`, `__project_theme_enabled`, etc.), chaque template dispose de **son propre flag d'activation par niveau** défini dans `attributes_config.adoc` :

[source,asciidoc]
----
// system/attributes/attributes_config.adoc

// Flags niveau global
:__global_template_label_question_enable: 0
:__global_template_label_answer_enable: 0
:__global_template_label_answer_available_enable: 0
:__global_template_label_teacher_note_enable: 0
:__global_template_todo_enable: 0

// Flags niveau projet
:__template_label_question_custom_enable: 0
:__template_label_answer_custom_enable: 0
:__template_label_answer_available_custom_enable: 0
:__template_label_teacher_note_custom_enable: 0
:__template_label_todo_custom_enable: 0
----

Tous sont à `0` par défaut : le framework utilise les templates système.

==== Résolution dans le resolver

Le resolver traite chaque template individuellement avec la cascade Système → Global → Projet. Voici la logique pour le template question (les 4 autres suivent le même schéma) :

[source,asciidoc]
----
// resolver

// --- question
:__template_label_question_file: {__system_template_question_file}            <1>
ifeval::[{__global_template_label_question_enable} == 1]
:__template_label_question_file: {__global_template_question_file}            <2>
endif::[]
ifeval::[{__template_label_question_custom_enable} == 1]
:__template_label_question_file: {__project_template_question_file}           <3>
endif::[]
----
<1> Par défaut, le template pointe vers le fichier système (`system/templates/`)
<2> Si le flag global est activé, le chemin est remplacé par celui du global (`global/templates/`)
<3> Si le flag projet est activé, le chemin est remplacé par celui du projet (`projects/{id}/templates/`)

==== Différence avec le thème et le docinfo

[cols="2,1,1,1"]
|===
| Caractéristique | Thème | Docinfo | Templates

| Granularité du flag
| 1 flag par niveau (global/projet/document)
| 1 flag par niveau
| **1 flag par template par niveau** (5 flags indépendants × niveaux)

| Niveaux de cascade
| Système → Global → Projet → Document
| Système → Global → Projet → Document
| **Système → Global → Projet** (pas de niveau document actuellement)

| Possibilité de mix
| Non (tout ou rien par niveau)
| Non (tout ou rien par niveau)
| **Oui** (on peut surcharger le label question au niveau global tout en gardant le label réponse système, puis un projet peut surcharger uniquement le label réponse)
|===

Cette granularité fine permet par exemple de personnaliser le format du label de question au niveau global pour tous les projets, tout en laissant chaque projet libre de personnaliser ses propres templates.

==== Les chemins sont prédéfinis dans `attributes_filesystem.adoc`

Chaque niveau dispose de ses propres chemins vers les fichiers de templates, définis dans `attributes_filesystem.adoc` :

[source,asciidoc]
----
// Niveau système
:__system_templates_folder: {__system_folder}templates/
:__system_template_question_file: {__system_templates_folder}label_question_template.adoc
:__system_template_answer_file: {__system_templates_folder}label_answer_template.adoc
// ... (idem pour les 3 autres)

// Niveau global
:__global_templates_folder: {__global_folder}templates/
:__global_template_question_file: {__global_templates_folder}label_question_template.adoc
:__global_template_answer_file: {__global_templates_folder}label_answer_template.adoc
// ... (idem pour les 3 autres)

// Niveau projet
:__project_templates_folder: {__current_project_folder}templates/
:__project_template_question_file: {__project_templates_folder}label_question_template.adoc
:__project_template_answer_file: {__project_templates_folder}label_answer_template.adoc
// ... (idem pour les 3 autres)

// Niveau document
:__document_templates_folder: {__current_document_folder}templates/
:__document_template_question_file: {__document_templates_folder}label_question_template.adoc
:__document_template_answer_file: {__document_templates_folder}label_answer_template.adoc
// ... (idem pour les 3 autres)
----

Les chemins document existent dans le filesystem mais ne sont **pas encore exploités par le resolver** pour les templates (qui gère système → global → projet pour l'instant). L'infrastructure est prête pour un futur niveau de surcharge document.

==== Exemple concret : personnaliser le label de question

Pour qu'un projet utilise son propre label de question :

. Activer la config projet (`enable_config.adoc` → `:__configuration_enabled: 1`)
. Activer le flag du template dans `attributes_config.adoc` :
+
[source,asciidoc]
----
:__template_label_question_custom_enable: 1
----
. Créer le fichier `projects/{id}/templates/label_question_template.adoc` avec le contenu personnalisé, par exemple :
+
[source,asciidoc]
----
*Question {counter:__question_number} —*
----

Le resolver résoudra `__template_label_question_file` vers le fichier projet au lieu du fichier système. Les autres templates (réponse, note, etc.) resteront ceux du système.

==== Exposition à l'utilisateur via les alias

Les templates résolus sont exposés à l'auteur du document via des alias simplifiés dans `attributes_alias.adoc` :

[source,asciidoc]
----
:__label_question: {__template_label_question_file}
:__label_answer: {__template_label_answer_file}
:__label_answer_hidden: {__template_label_answer_available_file}
:__label_teacher_note: {__template_label_teacher_note_file}
:__label_todo: {__template_label_todo_file}
----

L'auteur utilise ensuite ces alias dans son document via un `include` :

[source,asciidoc]
----
\include::{__label_question}[]
Quel est le rôle d'un serveur DNS ?

\include::{__label_answer}[]
Un serveur DNS traduit les noms de domaine en adresses IP.
----

Sans se soucier de savoir si le template provient du système ou du projet — le framework a déjà résolu le bon chemin.

== Phase d'alias

En dernière étape, `attributes_alias.adoc` expose des noms simplifiés destinés aux utilisateurs finaux dans leurs documents :

[source,asciidoc]
----
:__show_correction: {__render_show_correction}
:__question_number: {__render_question_start}
:__label_answer: {__template_label_answer_file}
:__label_question: {__template_label_question_file}
----

L'utilisateur utilise `{__show_correction}` dans son document sans se soucier de quel niveau a défini la valeur.

== Points clés à retenir

. **Le verrou est réinitialisé** (`__configuration_enabled: 0`) avant chaque niveau — les niveaux global, projet et document sont totalement indépendants les uns des autres.
. **Le niveau global permet de personnaliser sans toucher au système** — il s'intercale entre le système et les projets, offrant des valeurs par défaut organisationnelles qui s'appliquent à tous les projets.
. **Seuls les attributs redéfinis sont surchargés** — un fichier global qui ne définit que 3 attributs ne modifie que ces 3 valeurs ; tous les autres gardent leur valeur système. Idem pour les niveaux projet et document.
. **Le resolver ajoute une deuxième couche de cascade** pour les chemins (dossiers thème, docinfo, templates) qui dépendent à la fois de la valeur de l'attribut ET du dossier source.
. **Les alias isolent l'utilisateur** de la mécanique interne — il manipule des noms stables quel que soit le niveau qui a défini la valeur.
